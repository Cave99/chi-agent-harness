# CHI Explorer — Documentation

## Overview
CHI Explorer is a conversational call analysis tool that allows users to ask natural language questions about call center data and receive AI-powered answers, including charts, statistics, and executive summaries. It replaces the manual process of writing SQL, building prompts, and processing batch outputs.

The system is currently in a **"Home Build"** state, optimized for local development using OpenRouter for LLM inference and synthetic data for transcripts and metadata. It is designed with a stubbed architecture that allows for a seamless transition to AWS Bedrock and a live production database.

---

## Architecture & Pipeline
The application follows an 8-step pipeline to transform a user's question into a comprehensive analytical report.

1.  **User Input:** The user asks a question (e.g., "Are agents following the approved script for broadband upgrades?") via the React frontend.
2.  **Analysis Planning (Business Agent):** The **Business Agent** analyzes the question and generates a plan consisting of:
    -   A **SQL WHERE clause** to filter the relevant calls.
    -   A **novel system prompt** tailored to extract specific data from transcripts.
    -   A **field manifest** defining the structure and types of the extracted data.
3.  **Approval Gate:** The user reviews the plan, including the matched call count, estimated cost, and a preview of the prompt and sample transcripts. The user can refine the plan using AI or manual edits before approving.
4.  **Batch Processing (Stubbed):** Upon approval, the system fetches metadata for the matched calls and dispatches a batch job. In the current build, this is a stub that generates synthetic LLM outputs based on the field manifest.
5.  **Parsing & Enrichment:** The **Parser** processes the (synthetic) JSONL batch output, handles malformed lines using `jsonrepair`, and enriches the data with call metadata (agent name, team, etc.). It produces a structured summary dictionary and a list of enriched records.
6.  **Data Analysis (Data Agent):** The **Data Agent** receives the question, field manifest, and a sample of the records. It generates a Python script designed to run in a restricted sandbox. This script uses Pandas to perform aggregations and calculations over the full dataset.
7.  **Visualization & Execution (Sandbox):** The **Sandbox** executes the generated Python script, providing it with the full dataset as a Pandas DataFrame. The script outputs:
    -   A concise natural language **answer**.
    -   A list of **Recharts JSON configurations** for interactive charts.
8.  **Executive Summary (Vision Agent):** The **Vision Agent** receives the Recharts JSON specs and the original question. It writes a "numbers-first" executive summary, lists key findings, and suggests follow-up questions for further exploration.

## Core Components

### 1. Agents (`chi-explorer/agents/`)
-   **`business_agent.py`**: The "brain" that plans the analysis. It translates high-level questions into technical specs (SQL, Prompts, Manifest).
-   **`data_agent.py`**: The "data scientist" that writes Python/Pandas code to analyze the extracted data and define visualizations.
-   **`vision_agent.py`**: The "manager" that synthesizes the final report from the chart data and original context.

### 2. Pipeline (`chi-explorer/pipeline/`)
-   **`provider.py`**: An abstraction layer for LLM providers (OpenRouter vs. Bedrock). All agent calls pass through here.
-   **`db.py` (Stubbed)**: Handles SQL queries. Currently returns synthetic counts and metadata for testing.
-   **`batch.py` (Stubbed)**: Manages batch LLM jobs. Currently triggers the synthetic data generator.
-   **`parser.py`**: A schema-driven parser that converts raw LLM outputs into structured data based on the field manifest.
-   **`sandbox.py`**: A secure execution environment for the Python scripts generated by the Data Agent. It uses AST linting to prevent unauthorized operations (e.g., file I/O, network calls).
-   **`synthetic.py`**: Generates realistic fake data for testing the pipeline without incurring real LLM costs.

### 3. Session Management (`chi-explorer/session/`)
-   **`state.py`**: Manages in-memory session state, including message history, current job details, and analysis results.

## Frontend
The frontend is a modern React application built with **Vite**, **Tailwind CSS**, and **Recharts**. It communicates with the FastAPI backend via REST and Server-Sent Events (SSE) for real-time streaming of LLM responses and pipeline status updates.

-   **Approval Gate:** A critical UI component that allows for human-in-the-loop validation of AI-generated plans.
-   **Interactive Charts:** Uses `recharts` to render data-driven visualizations produced by the Data Agent.
-   **Streaming UI:** Provides immediate feedback as agents "think" and work through the pipeline.

## Development Guide

### 1. Modifying Agents
Each agent has a dedicated file in `chi-explorer/agents/`. To update an agent's behavior:
-   **System Prompts:** Modify the `SYSTEM_PROMPT` constant at the top of the agent's file.
-   **Output Structure:** If you change the expected JSON structure of an agent's output, ensure you update the parsing logic in the corresponding agent or the `pipeline/parser.py` if it affects the field extraction.

### 2. Updating the Pipeline
-   **New Field Types:** To support a new data type (e.g., "geospatial"), update `pipeline/parser.py` to handle the aggregation logic and `pipeline/synthetic.py` to generate sample data for it.
-   **Sandbox Security:** If you need to add a new library for the Data Agent to use, update the `ALLOWED_IMPORTS` in `pipeline/sandbox.py`.

### 3. Testing
The project includes several test scripts to validate individual components:
-   `test_provider.py`: Verifies LLM connectivity.
-   `test_parser.py`: Tests the JSONL parser against synthetic data.
-   `test_business_agent.py`: Validates the planning phase (requires an API key).

### 4. UI Customization
The frontend is located in the `frontend/` directory.
-   **Styling:** Follow the guidelines in [DESIGN.md](./DESIGN.md) for color, typography, and spacing.
-   **Components:** Add or modify React components in `frontend/src/App.tsx`.
-   **Charts:** The UI uses `recharts`. If you need a new chart type, ensure the `Data Agent` in the backend is updated to produce the corresponding JSON configuration.

## Design System: "Editorial Energy"
The project follows a specific design philosophy documented in [DESIGN.md](./DESIGN.md), focusing on:
-   **Radiant Orange** (`#ac3500`) as the primary brand color.
-   **Ambient Tonal Layering** instead of hard borders or heavy shadows.
-   **Authoritative Typography** (Plus Jakarta Sans and Be Vietnam Pro).
-   **Asymmetric Layouts** to create a premium, "editorial" feel rather than a standard utility dashboard.
