# CHI Explorer — Porting Guide (AWS Bedrock & Live DB)

This document outlines the steps required to transition CHI Explorer from the current "Home Build" (OpenRouter + SQLite stubs) to a production-ready system utilizing **Amazon Bedrock** and a **live SQL database**.

---

## 1. Environment Configuration

The first step is to update the `.env` file (or `config.py`) to enable the production-ready features.

```bash
# .env updates
INFERENCE_PROVIDER=bedrock
DB_ENABLED=true
BATCH_ENABLED=true

# AWS Credentials (ensure these are available in your environment)
AWS_REGION=us-east-1
# AWS_ACCESS_KEY_ID=...
# AWS_SECRET_ACCESS_KEY=...
```

---

## 2. Implementing Amazon Bedrock Support

The inference abstraction layer is located in `chi-explorer/pipeline/provider.py`. Currently, the Bedrock functions raise `NotImplementedError`.

### 2.1 Implementation Checklist:
-   **Add `boto3`:** Install the AWS SDK for Python (`pip install boto3`).
-   **Initialize Client:** In `provider.py`, initialize a Bedrock Runtime client.
-   **Implement `_bedrock_chat`:** Use the `converse` API for standard requests.
-   **Implement `_bedrock_stream`:** Use the `converse_stream` API for streaming responses to the frontend.

### Example Snippet (`provider.py`):
```python
import boto3

bedrock = boto3.client("bedrock-runtime", region_name=config.AWS_REGION)

def _bedrock_chat(messages, system, model):
    response = bedrock.converse(
        modelId=model,
        messages=messages,
        system=[{"text": system}]
    )
    return response["output"]["message"]["content"][0]["text"]
```

## 3. Implementing Bedrock Batch Inference

Large-scale transcript analysis (e.g., hundreds or thousands of calls) should be handled via the Bedrock Batch Inference API to minimize latency and cost.

### 3.1 Implementation Checklist (`pipeline/batch.py`):
-   **S3 Integration:** Batch inference requires input files to be uploaded to S3 and output to be written back to S3.
-   **Manifest Generation:** Generate the required JSONL manifest where each line is a model invocation request.
-   **Job Dispatch:** Use `bedrock.create_model_invocation_job()`.
-   **Polling:** Implement the `poll()` function to check the status of the job and download results from S3 once complete.

## 4. Connecting to a Live Database

The database logic is encapsulated in `chi-explorer/pipeline/db.py`.

### 4.1 Implementation Checklist:
-   **Choose a Library:** Use `psycopg2` for PostgreSQL or `SQLAlchemy` for a more flexible ORM-based approach.
-   **Update Connection Logic:** Replace the current SQLite stubs with real connection pooling.
-   **Schema Mapping:** Ensure the `business_agent` is aware of the production schema (update its `SYSTEM_PROMPT` in `agents/business_agent.py` if column names differ).
-   **Dialect Compatibility:** Verify that the `where_clause` generated by the Business Agent is compatible with your production database's SQL dialect.

### Example Snippet (`db.py`):
```python
import psycopg2
from psycopg2.extras import RealDictCursor

def get_connection():
    return psycopg2.connect(config.DATABASE_URL)

def count_calls(where_clause):
    with get_connection() as conn:
        with conn.cursor() as cur:
            query = f"SELECT COUNT(*) FROM calls WHERE {where_clause}"
            cur.execute(query)
            return {"count": cur.fetchone()[0], "stubbed": False}
```

## 5. Security & Scaling Considerations

-   **IAM Roles:** When deploying to AWS (e.g., EC2, ECS, or Lambda), use IAM roles instead of hardcoded access keys.
-   **Sandbox Hardening:** In a production environment, consider moving the `sandbox.py` execution to a serverless function (AWS Lambda) or a dedicated Docker container to provide stronger isolation for generated Python code.
-   **State Persistence:** Replace the current in-memory `session/state.py` with a persistent store like **Redis** or **DynamoDB** to support multi-user sessions and horizontal scaling.
-   **VPC Connectivity:** Ensure the CHI Explorer backend has the necessary VPC permissions and security group rules to access your production database and S3 buckets.
